## 🔴 Production-blocking bugs

### B1. Chain hangs in diff/judge under Cloud Run's async runtime
**Severity:** blocker for D9 video demo.

**Repro:** Upload any two PDFs through `/`; chain enters `running_chain` and stays there indefinitely. In this run the chain produced 19 `agent_runs` rows up to `09:30:37` UTC (4 lens × 2 clauses = 8 decompose calls + 11 map calls) and then went silent. No new rows for 12+ minutes. No exception traces in logs after that point. Only noise is benign aiohttp `Task was destroyed but it is pending!` cleanup warnings from the `google-genai` async client.

**Likely cause:** `app/runners.py::_run_in_fresh_loop` spawns a fresh asyncio loop per `run_agent` call. `app/sub_agents/diff.py` and `judge.py` run those through a `ThreadPoolExecutor(max_workers=8)`. Under uvicorn's worker process this creates a pile-up where the aiohttp connector from one fresh loop is closed by a different loop on a different worker thread — the runtime error we saw (`got Future attached to a different loop`) leaves a future pending forever. The thread holds the GIL but never returns, the `ThreadPoolExecutor.map` never completes, the parent `run_chain` blocks forever. Locally this doesn't show because `python -m app.chain` is single-threaded under one loop.

**Fix candidates (small → big):**
1. **Quick** — set `CURATOR_MAP_CONCURRENCY=1` and `CURATOR_DIFF_CONCURRENCY=1` on the deployed Cloud Run service. Trades the parallelization win for stability. Wall-clock goes back to ~15 min per chain.
2. **Better** — rewrite `_run_in_fresh_loop` so each ThreadPoolExecutor worker keeps its own pinned event loop for its lifetime, plus a wrap-and-close that doesn't await async cleanup in the wrong loop. ~30 LOC change in `app/runners.py`.
3. **Best** — replace the threaded fan-out with `asyncio.gather` inside one shared loop. Removes the cross-loop entirely.

For D9: recommend **Fix 1 deployed immediately** to unblock the video, then **Fix 2** as a follow-up before final submission.

### B2. `pipeline_run_id` is NULL on every `agent_runs` row on Cloud Run
**Severity:** observability-blocker for any "the agent improved itself" narrative.

**Repro:** `SELECT DISTINCT pipeline_run_id FROM agent_runs` returns only NULL. `begin_pipeline_run()` writes to a `ContextVar` and a module-level mirror; neither propagates through the `ThreadPoolExecutor` workers nor through the background thread that runs the chain. Without `pipeline_run_id`, you cannot group rows by run, and the Agent Optimizer can't reliably bucket prompt iterations by call.

**Fix:** pass `pipeline_run_id` explicitly into the worker `_run_one` closures of `map_.py`, `diff.py`, and the lens panels — bind it at submit time and forward to `run_agent`. ~10 LOC change touching 3 files. Also patch `app/observability/run_log.py::log_run` to accept an explicit `pipeline_run_id` kwarg that overrides the ContextVar read.

## 🟠 UX / wiring bugs

### U1. Nav link "A2A card" → 404
**Where:** all pages, `app/templates/base.html`.

The nav has `<a href="/.well-known/agent.json">A2A card</a>`. That path returns 404. The actual mount (a2a-sdk's spec path) is `/a2a/app/.well-known/agent-card.json`.

**Fix:** change the href to `/a2a/app/.well-known/agent-card.json`.

### U2. Analyse header says "0 clauses, four-lens debate in flight"
**Where:** `app/templates/analyse.html`.

The header always shows "0 clauses" even after Doc AI has finished parsing and Spanner has the chunks. The status fragment never updates `clause_count` on the parent page — only the `<p>Status:</p>` strong tag updates. Misleading.

**Fix:** include clause count in the polled status fragment, or compute server-side and stop hard-coding `0` in the heading.

### U3. Inbox rows have no "process this" CTA
**Where:** `app/templates/inbox.html`.

The Discovery Inbox shows 30 SEBI items, all with `status: new`, all with `Impact: —`. There is no per-row button to send a specific item through the chain. The only path is the page-level "Poll RBI now" (triggers another discovery cycle, not a chain run). The Pub/Sub subscriber is supposed to auto-trigger chain runs on new items, but in this test the items have sat in `new` status without firing — confirming that the subscriber `CURATOR_DISCOVERY_SUBSCRIBE=1` either isn't pulling or the per-item chain trigger is wired to a TODO.

**Fix:** add a per-row "Analyse" button that POSTs the item's URL (or item_hash) into the chain. Separately, audit the subscriber `app/discovery/subscriber.py` — there are two `TODO(D6)` markers per the hygiene-sweep report that may be exactly this.

### U4. Hard-coded "RBI" copy on a page that polls SEBI
**Where:** `app/templates/inbox.html`, headline copy reads "Curator polls RBI's public RSS every 30 minutes." The actual source is SEBI (RBI's RSS is broken — documented in `RECONCILER_AUDIT_2026-06-02.md` and `curator-stage-2-resume` memory). Misleading for a judge who clicks "Inbox" and sees only SEBI sources.

**Fix:** swap "RBI" → "RBI/SEBI" in the headline; or rename the manual button "Poll RBI now" → "Poll regulators now".

### U5. No favicon (cosmetic)
`GET /favicon.ico → 404`. Adds one console error on every page load. Fix by serving `app/static/favicon.ico`.
